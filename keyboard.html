Keyboard Compatibility
Web Accessibility Perspectives

Introduction

Web accessibility is essential for people with disabilities but useful for many more. Learn about some of the accessibility considerations and how they benefit everyone in different situations.

About Keyboard Compatibility

All functionality needs to work with the keyboard. That is, moving between controls using the tab key and other shortcuts, and typing into input fields. This includes moving between any links, buttons, forms, and other controls. Many websites support use with a mouse only. For example, they do not allow you to type in a date as well as to select it from a pop-up calendar.

Who depends on this feature?

  • People with physical disabilities who cannot use the mouse;
  • People who are blind, and cannot see the mouse pointer on the screen;
  • People with chronical conditions, such as repetitive stress injuries (RSI), who should not use a mouse.

What are the additional benefits?

  • Content works for people with temporary limitations, such as a broken arm or broken mouse;
  • Content works in more situations, such as with on-screen keyboards and terminals without a mouse.

What needs to happen for this to work?

Native HTML controls, like links, buttons, and form elements, work with the keyboard by default. Problems arise when custom-made controls, CSS styles, and scripts that control interaction break this compatibility. Also the navigation between multiple controls could cause barriers. For example, when the order of the controls in the source code does not match the visual presentation, or when there are too many controls without a way to jump between groups of related controls. Such issues occur particularly in forms, menus, and applications with many controls.
